Part 4 — The five mental models

Internalize these and the rest is detail. This is the gap between a daily user and a developer:

  1. Everything is a table. Incidents, your app, and even configuration itself (business rules, UI policies, ACLs) are all just records in tables. "Configuring ServiceNow" = inserting/updating records. Biggest single unlock.
  2. Client vs. server is the master split. Some code runs in the browser (g_form, g_user), some on the server (GlideRecord, gs). Knowing where a script runs answers half the scripting questions on the exam.
  3. GlideRecord is the ORM. It's how server-side JavaScript queries/creates/updates/deletes records. You'll write it constantly.
  4. ACLs are evaluated, not assumed. Access = role + condition + script, evaluated most-specific-first. The 20% security domain is mostly "can this user do this operation here, and why."
  5. A scoped application is your code in its own namespace (x_yourapp_…), sandboxed from the global platform. CAD is fundamentally about building scoped apps.

5.1 GlideRecord — querying on the server

The workhorse. Run these in your PDI under All → Scripts - Background (a server-side scratchpad). gs.info() writes to the script output / system log.

Scripts - Background (server) 
// Find every open P1 incident and log it.
var gr = new GlideRecord('incident');
gr.addQuery('priority', 1);          // priority = 1 (Critical)
gr.addQuery('active', true);         // still open
gr.query();                          // run it against the database
while (gr.next()) {                  // step through each matching row
  gs.info('Open P1: ' + gr.getValue('number') + ' — ' + gr.getValue('short_description'));
}

To grab one record by a unique value, use get() — it returns a boolean and positions you on the row:

var gr = new GlideRecord('incident');
if (gr.get('number', 'INC0010001')) {   // true if found
  gr.setValue('priority', 1);
  gr.update();                          // persist the change
}
Three traps the exam loves (1) query() returns many rows (loop with next()); get() returns one (a boolean). (2) Use getValue('field') to read the raw stored value — reading gr.field directly returns an object that can bite you in comparisons. (3) gr.setWorkflow(false) before an update suppresses business rules for that operation — powerful, and a classic "what does this do" question.

5.2 Business Rule — server logic on a database event

A Business Rule runs on the server around a database operation. The four "when" options are the whole game:

WhenRunsTypical use
beforeBefore the record is writtenSet/validate field values on the same record (no update() needed — it's about to save)
afterAfter the record is writtenTouch other records / send notifications
asyncAfter commit, in the backgroundHeavy work that shouldn't slow the user's save
displayWhen the form loadsPass server data to client scripts via the g_scratchpad
Business Rule · before · table: incident (server) 
(function executeRule(current, previous /*, gs */) {
  // 'current' is the record being saved; 'previous' is its prior state.
  if (current.priority == 1) {
    current.work_notes = 'Auto-flagged as P1 by a business rule.';
  }
  // No current.update() here — a 'before' rule saves automatically.
})(current, previous);
The recursion trap Calling current.update() inside an after rule on the same record re-triggers the rules → infinite loop. In a before rule you never need it. This distinction is prime exam material.

5.3 Client Script — reacting in the browser

Runs in the user's browser via g_form. The four types are onLoad, onChange, onSubmit, onCellEdit. Here's an onChange on the Priority field:

Client Script · onChange · field: priority (browser) 
function onChange(control, oldValue, newValue, isLoading) {
  if (isLoading || newValue === '') return;   // skip the initial form load
  if (newValue == '1') {                       // Priority set to Critical
    g_form.setMandatory('work_notes', true);
    g_form.addInfoMessage('P1 selected — work notes are now required.');
  }
}
Client Script vs. UI Policy — a favourite question If all you need is show / hide / make-mandatory based on a field value, use a UI Policy (no code, easier to maintain). Reach for a Client Script only when you need logic a UI Policy can't express (calculations, calling the server, conditional messages). "Prefer the lower-code tool" is the platform's whole philosophy.

5.4 Script Include + GlideAjax — the client asks the server

A Client Script can't query the database directly (it's in the browser). When it needs server data, it calls a Script Include via GlideAjax. This pairing shows up constantly on the exam.

Server A client-callable Script Include. It extends AbstractAjaxProcessor and reads parameters the client sent:

Script Include · client callable · IncidentUtils (server) 
var IncidentUtils = Class.create();
IncidentUtils.prototype = Object.extendsObject(global.AbstractAjaxProcessor, {

  getOpenCountForCaller: function() {
    var caller = this.getParameter('sysparm_caller');   // value sent by the client
    var ga = new GlideAggregate('incident');            // efficient COUNT, no row loop
    ga.addQuery('caller_id', caller);
    ga.addQuery('active', true);
    ga.addAggregate('COUNT');
    ga.query();
    return ga.next() ? ga.getAggregate('COUNT') : '0';  // returned to the client
  },

  type: 'IncidentUtils'
});

Browser A Client Script on the Caller field calls it and uses the answer:

Client Script · onChange · field: caller_id (browser) 
function onChange(control, oldValue, newValue, isLoading) {
  if (isLoading || newValue === '') return;

  var ga = new GlideAjax('IncidentUtils');              // name of the Script Include
  ga.addParam('sysparm_name', 'getOpenCountForCaller'); // method to run
  ga.addParam('sysparm_caller', newValue);              // arg → this.getParameter(...)
  ga.getXMLAnswer(function(answer) {                    // async — runs when the server replies
    if (parseInt(answer, 10) > 5) {
      g_form.addInfoMessage('Heads up: this caller already has ' + answer + ' open incidents.');
    }
  });
}
Details that trip people up (1) The custom param names must start with sysparm_. (2) In a scoped app you extend global.AbstractAjaxProcessor (cross-scope), as shown. (3) GlideAjax is asynchronous — your code continues before the answer arrives, so the response is handled in the callback, not on the next line.

5.5 Flow Designer — when not to script at all

Not everything should be code. Flow Designer builds multi-step automation (trigger → conditions → actions) with almost no scripting, and it's far easier for the next person to read and maintain.

The decision the exam wants you to make Reach for Flow Designer for process automation (when X happens, do these steps — approvals, notifications, task creation). Reach for a Business Rule for data-level enforcement that must fire on every write regardless of channel. "Use the lowest-code tool that fully does the job" is the right answer far more often than new developers expect.

Part 6 — Build it (your tangible win)

Reading scripting without running it won't pass CAD. In your PDI, do this end-to-end — it exercises four of the five script types at once:

  1. Open Scripts - Background and run the GlideRecord query from 5.1. Confirm P1s print to the output.
  2. Create the before Business Rule from 5.2 on incident. Open a new incident, set Priority = 1, save, and confirm the work note appears.
  3. Add the onChange Client Script from 5.3. Watch work notes become mandatory the instant you pick P1 — no save needed (that's the client/server difference made visible).
  4. Create the Script Include + GlideAjax pair from 5.4. Pick a caller with a few incidents and watch the count message appear.
  5. Now reflect: which of these had to be server-side, and which had to be client-side? If you can explain why for each, you've got the core of the Automation domain.

Check your understanding

Answer before scrolling — retrieval is what makes it stick.

A Client Script needs the count of a caller's open incidents from the database. What's the correct mechanism?

Client Scripts run in the browser and can't touch the database directly. They call server-side Script Includes through GlideAjax. GlideRecord is server-only.

You only need to hide a field when "Category" is "Hardware." Which tool is the best fit?

Show/hide/mandatory based on a field value is exactly what a UI Policy does — no code, easy to maintain. Prefer it over a Client Script whenever it can do the job.

Inside a before Business Rule you set current.state = 2. What else must you do to persist it?

A before rule runs before the write, so your change rides along with the save. Calling current.update() here (or in an after rule on the same record) risks recursion.

Which CAD domain carries the most weight — and is the biggest gap for someone coming from daily use?

Managing Applications (scoped apps, Studio, update sets, versioning) is 25% and almost entirely new to a user who hasn't built before. Start there.
